Opening — 30 seconds

I'm going to demo Event Ledger — a two-service .NET system that ingests financial transaction events from upstream systems that may send duplicates or deliver events out of order. The Event Gateway is the public API — validation, idempotency, audit log. The Account Service owns balances and transactions. Each has its own SQLite database. I'll walk through the happy path, idempotency, out-of-order handling, validation errors, and show the resulting balances.

A Happy path — ~3 min

0a — Gateway health
GET http://localhost:8080/health
Both services expose health checks that verify database connectivity. Gateway returns 200 when SQLite is reachable.
200 OKstatus: Healthy
0b — Account Service health
GET http://localhost:8081/health
Account Service is on 8081 for demo and debugging. In production, only the Gateway would be public — clients never post events directly to Account.
200 OK
1 — CREDIT $150 (creates account)
POST /events · eventId evt-credit-1001
I'm submitting a credit of one hundred fifty dollars. No separate create-account call — the account is auto-created on the first transaction. The Gateway validates, checks idempotency, calls Account Service, then persists the audit record only after Account succeeds.
201 Createdstatus: Applied
2 — DEBIT $40
POST /events · type DEBIT · amount 40.00
Now a debit of forty dollars. Amount is always positive in the API — the type field determines credit versus debit.
201 Created
3 — Idempotency replay ⭐
Resubmit the exact same CREDIT from step 1
Upstream systems retry — so I'll resubmit the same eventId. Watch the response.
200 OK Idempotency-Replay: true same body as step 1
Same event returned, no duplicate transaction, balance unchanged. The Gateway short-circuited — Account Service wasn't called again.
4 — Get event by ID
GET /events/evt-credit-1001
Reads hit the Gateway's local audit database only — no Account Service call.
200 OK
5 — List events for account
GET /events?account=acct-demo-001
Events are listed in chronological order by eventTimestamp, not arrival order.
200 OK2 events: credit 16:00, debit 16:05
6 — Balance ⭐
GET /accounts/acct-demo-001/balance
Balance lives in Account Service — sum of credits minus debits.
200 OKbalance: 110.00
One hundred ten — exactly what we'd expect.
7 — Account detail
GET /accounts/acct-demo-001
Account detail includes metadata and recent transaction history.
200 OK

B Out-of-order arrival — ~2 min

Next I'll prove out-of-order tolerance. I'll submit three events in reverse chronological order — latest timestamp first, earliest last.
8a — T3: CREDIT +$50 @ 18:00 (submit first)
POST /events · evt-ooo-t3 · acct-demo-out-of-order
201 Created
8b — T2: DEBIT $30 @ 17:00
POST /events · evt-ooo-t2
201 Created
8c — T1: CREDIT +$100 @ 16:00 (submit last)
POST /events · evt-ooo-t1
I submitted T3, then T2, then T1 — reverse order. Balance math is commutative: credits minus debits, regardless of arrival order.
201 Created
8d — Balance ⭐
GET /accounts/acct-demo-out-of-order/balance
200 OKbalance: 120.00100 − 30 + 50
One hundred twenty — correct even though events arrived out of order.
8e — Event list (ordering)
GET /events?account=acct-demo-out-of-order
The listing is sorted by eventTimestamp, not submission order. That's the audit view clients expect.
200 OKT1 → T2 → T3 by timestamp

C Validation errors — ~1 min

Finally, validation — fail fast with no side effects.
9 — Negative amount
POST /events · amount: -5.00
Negative amount rejected. Nothing persisted, Account Service never called. Point out the traceId field in the ProblemDetails JSON — clients can correlate errors with logs.
400 Bad RequestProblemDetailstraceId in body

POST /events · type: TRANSFER
Only CREDIT and DEBIT are accepted. Clear error, no partial state.

E Observability — ~2 min (optional)

No Jaeger/Grafana in this project — observability is shown via structured logs, traceId in errors, and OpenTelemetry console export.

E0 — Open log tail (prep)
Second terminal
docker compose logs -f event-gateway account-service
I'll tail structured JSON logs from both services. Every request gets a trace ID that propagates Gateway to Account via the W3C traceparent header.
E1 — Distributed trace in logs ⭐
POST /events · eventId evt-obs-trace-001
Submitting a new event — watch both log streams. The same TraceId appears on Gateway and Account Service lines. Gateway logs Event applied; Account logs the transaction. OpenTelemetry also emits span activity to the console.
201 Created Gateway log: TraceId + ServiceName EventGateway Account log: same TraceId

Example log fields: TraceId, ServiceName, EventId, AccountId. Custom metric eventledger.events.processed increments on success.

E2 — traceId in error response ⭐
POST /events · amount -1.00 (or re-use step 9)
Validation errors return RFC 7807 ProblemDetails with a traceId extension. Support teams can take this ID and find the matching log line without guessing.
400 Bad Request "traceId": "…" in JSON body

Closing — 30 seconds

To summarize: dual-layer idempotency on eventId; persist after downstream success so we don't store orphan events; out-of-order tolerance via commutative balance and chronological listing; graceful degradation — GET /events works from Gateway local DB even if Account is down; layered resiliency — inbound rate limiting and bulkhead, outbound Polly circuit breaker with retry and timeout; OpenTelemetry trace propagation and structured JSON logs. The repo has automated tests covering these scenarios. Happy to go deeper on any design decision.

Follow-up Q&A

QuestionAnswer
Why two services?Gateway = ingestion + audit boundary; Account = financial source of truth
Why persist after Account succeeds?Avoid orphan events; client retries safely via idempotency
Why allow negative balance?Simpler than holds; commutative sum avoids ordering bugs
Retry on POST?Yes — 3 retries with jitter; safe because Account idempotency keys on eventId
What if Account is down?POST → 503, event not stored; GET /events still serves local data
